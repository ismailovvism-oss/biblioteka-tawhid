# SPEC v1 — Параллельная читалка Markdown-книг («Tawfiq al-Mannan reader»)

> Этот документ — **спецификация для реализации**, а не код. Код пишет Claude Code
> («Клудод»). Здесь зафиксированы контракт данных, модель, режимы, фазы и правила.
> Документ — источник истины: при расхождении кода и спеки правится либо код, либо
> (осознанно) спека, но не молча.

---

## 1. Что это

Веб-приложение для **параллельного чтения** книги в двух языках (оригинал + перевод),
выровненных по смысловым **секторам**. Первая книга — «Тауфик аль-маннан», т. 2
(арабский ↔ русский), но приложение — **обёртка над любой парой языков**: оно ничего не
знает о конкретной книге и работает с любым контентом, соблюдающим Контракт (раздел 3).

Две цели пользователя:
1. **Параллельное чтение** — оригинал и перевод рядом/чередуясь.
2. **Изучение языка** — видеть строку оригинала и сразу её перевод.

## 2. Стек и ограничения

- **Чистый vanilla JS + HTML + CSS.** Без фреймворков, без сборщиков. Один-два .js-файла.
- **Статический хостинг (Netlify).** Бэкенда в v1 нет.
- **GitHub-репозиторий**, разработка с разных устройств. Структура — обычный статический сайт.
- **Контент НЕ бандлится в код.** Лежит отдельными папками, грузится в рантайме через `fetch`.
- **Мобайл-фёрст.** Основной сценарий — телефон, вертикально. Альбомная/планшет — вторично.
- **НЕ использовать** `localStorage`/`sessionStorage`-зависимый функционал как критический
  (Netlify ок, но настройки храним в `localStorage` — это допустимо для статики; в артефактах
  Claude.ai это запрещено, но здесь обычный сайт, так что можно).
- Никаких готовых markdown-рендереров для тела: **парсер сносок/якорей пишется свой**
  (стандартные рендереры якоря-комментарии и сектора корректно не обработают). Допустимо
  использовать лёгкий markdown→HTML только для **inline-разметки внутри сектора** (жирный,
  курсив, кавычки), но разбиение на сектора/сноски/страницы — собственное.

## 3. КОНТРАКТ КНИГИ (ядро переиспользования)

Любая книга, чтобы открыться в приложении, обязана соблюдать следующее.

### 3.1 Структура папок
```
/books/<bookId>/
    ar/NN.md          # оригинал, по главам (NN — номер главы, напр. 07)
    ru/NN.md          # перевод, ТЕ ЖЕ имена файлов
    img/pNNN.jpg      # скан страницы; имя = номер страницы (см. 3.4). Опционально.
    book.json         # манифест книги (см. 3.5)
```
- Имена файлов глав в `ar/` и `ru/` **идентичны**.
- `img/` опциональна; если её нет — функция «показать скан» просто недоступна.

### 3.2 Сектор
Минимальная единица выравнивания. Обычно абзац или предложение.

- Каждый сектор предваряется якорем на **отдельной строке**:
  `<!-- s007 -->`
- Затем идёт текст сектора (может быть несколько строк до пустой строки).
- Сектора разделяются пустой строкой.
- Якорь `sNNN` — **уникален в пределах главы** и **совпадает** в `ar/NN.md` и `ru/NN.md`.
- Выравнивание пар идёт **по совпадению id**, НЕ по порядку.

> Почему HTML-комментарий: невидим в Obsidian-рендере (автор правит .md как обычный текст),
> тривиально парсится, отбрасывается при любом экспорте, не конфликтует с Markdown.

**Группы (1 оригинал ↔ несколько секторов перевода).** Если один арабский сектор
переведён двумя русскими, русские получают суффикс:
```
ar:  <!-- s050 -->            ru:  <!-- s050a -->
                                   <!-- s050b -->
```
Приложение группирует `s050a`, `s050b` под базовым id `s050` и показывает их как
**одну пару** (арабский сектор напротив склейки русских). Базовый id = id без хвостовой
буквы. Симметрично возможен обратный случай (`s050a/b` в `ar`, один `s050` в `ru`).

### 3.3 Сноска
- Текст сносок — **блоком в конце файла главы**.
- Якорь сноски: `<!-- fn073 -->` на отдельной строке перед текстом сноски.
- Ссылка на сноску внутри сектора — маркер вида `[^73]` (как в Markdown/Obsidian).
- Сноски **пер-язычные**: нумерация и текст у каждого языка свои, между языками НЕ
  спариваются. Авторские сноски-цитаты (в `ar`) и пояснения переводчика (только в `ru`)
  — разной природы и числа. Ссылка `[^N]` ведёт к сноске `fnN` **того же языка**; в
  показе одного языка сноски другого скрыты целиком. (Если у книги сноски всё же
  переводы друг друга — они просто покажутся двумя отдельными записями.)
- Многоабзацные сноски: абзацы внутри сноски разделяются пустой строкой; сноска
  заканчивается на следующем якоре `<!-- fnNNN -->` или конце файла.

### 3.4 Страница (для навигации и будущих сканов)
- Маркер: `<!-- p152 -->` на отдельной строке перед сектором.
- Ставится **ТОЛЬКО в `ar/`**. В `ru/` маркеров страниц НЕТ.
- Относится ко всем последующим секторам **до следующего** `<!-- pNNN -->`.
- Нумерация **сквозная по всему тому** (p1…p600). Значит `pNNN` глобально уникально.
- Парсер строит карту `sectorId → page` из `ar/`, затем **протягивает** page на русские
  сектора по совпадению id. Русский сектор без арабской пары останется без page —
  это **сигнал ошибки выравнивания** (см. 6, валидатор).

### 3.5 Манифест `book.json`
```json
{
  "bookId": "tawfiq-al-mannan-v2",
  "title": { "ar": "...", "ru": "Тауфик аль-маннан, т.2" },
  "languages": ["ar", "ru"],
  "rtl": ["ar"],
  "chapters": [
    { "file": "01.md", "title": { "ar": "...", "ru": "Вступление" } },
    { "file": "07.md", "title": { "ar": "...", "ru": "Защита Ибн Таймийи (2/3)" } }
  ],
  "hasImages": true,
  "imagePattern": "img/p{page}.jpg"
}
```
- `rtl` — список языков, рендерящихся справа-налево. Направление берётся **из языка
  контента**, не из позиции окна.
- `chapters` задаёт порядок и оглавление.

## 4. МОДЕЛЬ ДАННЫХ (ядро)

Парсер на входе берёт `ar/NN.md` + `ru/NN.md`, на выходе даёт **массив пар**:

```js
// одна глава →
[
  {
    id: "s007",          // базовый id
    page: 152,           // из ar, протянуто; null если неизвестно
    type: "text",        // "text" | "footnote"
    ar: "…html сектора…", // inline-markdown → html; null если нет
    ru: "…html…",        // null если нет
    refs: ["73","74"]     // id сносок, на которые ссылается этот сектор (из [^N])
  },
  …
]
```

- **Всё остальное в приложении — функции отрисовки над этим массивом.** Раскладки,
  видимость языков, сноски — это представление одной модели. Нельзя дублировать данные
  под разные режимы.
- Сноски — те же элементы массива с `type:"footnote"`, живут логически «в конце главы».

## 5. РЕЖИМЫ ОТОБРАЖЕНИЯ (состояние поверх ядра — НЕ отдельные архитектуры)

### 5.1 Базовая раскладка — ЕДИНЫЙ ПОТОК ПАР
Поток идёт сверху вниз: для каждой пары — оба языка вместе (по умолчанию оригинал сверху,
перевод снизу). Это и «билингва для учёбы», и (через 5.2) «чтение одного языка», и (через
5.4) «две колонки». **Отдельного механизма синхронного скролла двух независимых окон НЕТ** —
пара физически слита, синхронизировать нечего.

### 5.2 Видимость языков (3 состояния, одна кнопка-переключатель)
- `both` — оба члена пары видны (дефолт);
- `ar` — виден только арабский член каждой пары → сплошное чтение арабского;
- `ru` — виден только русский → сплошное чтение перевода.
Реализация: каждый член пары — блок с переключаемой видимостью. Скрытие — `display:none`.

### 5.3 Подсветка активной пары
Тонкая полоса/заливка у пары, ближайшей к центру вьюпорта. Помогает глазу держать
соответствие. Активная пара определяется по позиции относительно центра экрана.

### 5.4 Раскладка пары: вертикаль / горизонталь
- Вертикаль (телефон-портрет, дефолт): оригинал над переводом.
- Горизонталь (альбом/планшет): оригинал и перевод в две колонки (CSS `flex-direction`).
- Переключение **по ориентации экрана** автоматически + ручной тумблер в настройках.
- Это **CSS-переключение над тем же потоком**, не другая логика.

### 5.5 Направление текста
- Член пары на языке из `rtl` → `dir="rtl"`, иначе `dir="ltr"`.
- Привязано к языку контента, не к позиции. При смене языка в потоке dir пересчитывается.

### 5.6 Сноски — ДВЕ механики, переключатель в настройках
Источник один (`type:"footnote"` по id), различается только показ.
- **Инлайн-разворот (дефолт):** тап по `[^73]` → под текущим сектором раскрывается
  аккордеон с текстом сноски `fn073`; повторный тап — сворачивает. Разворачивается
  **в соответствии с текущей видимостью языков** (both → пара ar+ru).
- **Скачок-возврат:** тап по `[^73]` → `scrollIntoView` к сектору `fn073` (логически конец
  главы); запоминаем origin-id; у текста сноски — кнопка/маркер «вернуться» → скролл назад
  к origin и кратковременная подсветка маркера `[^73]`.
- Переключатель в настройках выбирает обработчик тапа.

### 5.7 Навигация
- **Оглавление** по `chapters` из манифеста.
- **Переход по странице:** ввод номера → прыжок к первому сектору с `page == N`.
- Текущая страница видна в UI (из `page` активной пары).

### 5.8 Скан страницы (если `hasImages`)
- Кнопка у активной пары «показать скан» → открывает `img/p{page}.jpg` (модал/оверлей,
  с зумом). Берётся по `page` сектора, не по id — устойчиво к дроблению.

## 6. ВАЛИДАТОР КОНТРАКТА (обязателен, дёшев, спасает книгу)

При загрузке главы приложение проверяет и **видимо сообщает** (не молча!):
- id сектора есть в `ar`, но нет пары в `ru` (или наоборот) → предупреждение со списком id;
- русский сектор без `page` (значит нет арабской пары) → тот же сигнал;
- ссылка `[^N]` в тексте без определения `fnN` в конце файла → битая сноска;
- определение `fnN` без единой ссылки `[^N]` → висячая сноша.
Эти проверки — главный страхователь от тихого рассинхрона. Вывести в отладочную панель
(скрытую за настройкой «режим проверки»), чтобы при разметке тома ты ловил ошибки сразу.

## 7. ФАЗЫ

**v1 (этот документ):** контракт, парсер, модель пар, единый поток, видимость языков (3),
раскладка вертикаль/горизонталь, направление RTL/LTR, сноски обеими механиками, оглавление,
навигация по страницам, скан-оверлей, валидатор. Контент — **голый текст** (inline-markdown).
Настройки в `localStorage` (тема, режим сносок, раскладка, видимость).

**v2 (НЕ делать в v1):**
- Дифференциация голосов (матн «Маджму'» / хадис / шарх) — потребует доп. разметки в .md
  (напр. класс на секторе через ещё один комментарий `<!-- voice:hadith -->`); отдельная фаза.
- Поиск (русский + арабский с нормализацией: снятие огласовок, варианты алифа, та-марбута).
- ИИ через OpenRouter (разбор грамматики/корня/и'раба выделенного арабского). Требует
  прокси для сокрытия ключа → первый повод для бэкенда (Netlify Function).
- Пословное выравнивание (тап по слову, а не сектору) — НЕ в контракте v1, крупная отдельная тема.

## 8. ПРАВИЛА ДЛЯ КЛУДОДА (Claude Code)

1. **Сначала ядро, потом представление.** Реализуй парсер + модель пар + валидатор и
   докажи на одной главе (07), что пары и страницы собираются верно, ДО любого UI.
2. **Одна модель — много рендеров.** Не дублируй данные под режимы. Режимы — функции над
   массивом пар.
3. **Контент в рантайме.** `fetch` из `/books/<id>/…`. Никакого вшивания текста в код.
4. **Не молчать об ошибках контракта.** Рассинхрон id/страниц — видимое предупреждение,
   не тихий сдвиг.
5. **Парсер сносок/якорей — свой.** Стандартный markdown-рендерер только для inline внутри
   сектора, и то опционально.
6. **Мобайл-фёрст, без зависимостей.** Если очень нужна библиотека — сперва обосновать здесь.
7. **Тестовые данные.** Создай `/books/_sample/` с 1–2 короткими главами (ar+ru, сноски,
   страницы, группа s050a/b) для разработки без реальной книги.
8. **README** с инструкцией запуска (любой статический сервер) и описанием структуры.
